## TC-011 — Pipeline completes: hypotheses generated

**Scope:** AI Pipeline  
**Spec ref:** §7 (steps 1–6), §4 (Hypothesis model)

### Preconditions
- Assessment in `analyzing` status
- Evidence documents exist (text extracted)
- Survey answers exist
- OpenAI API accessible

### Main Flow
1. Pipeline steps 1–6 execute sequentially
2. Step 1: context assembly from documents + survey
3. Step 2: knowledge base embedding query (pgvector)
4. Step 3: signal extraction (LLM returns structured `MatchedSignal[]`)
5. Step 4: deterministic scoring (scoring engine — pure)
6. Step 5: hypothesis generation (LLM returns `Hypothesis[]`)
7. Step 6: status → `hypotheses_ready`

### Expected Outcomes
- `THEN` Assessment status is `hypotheses_ready`
- `THEN` At least one Hypothesis row exists in DB
- `THEN` Each Hypothesis has: `pathologyId`, `title`, `explanation`, `confidence` (LOW/MEDIUM/HIGH), `status: "pending"`, `evidenceLevel`
- `THEN` `GET /assessments/:id/hypotheses` returns the hypotheses
- `THEN` No hypothesis has `confidence` expressed as a percentage (only LOW/MEDIUM/HIGH enum)

### Pipeline Step Contracts
- Step 3 (signal extraction) uses `MatchedSignal[]` interface — `signalId`, `source`, `excerpt`, `chunkId`
- Step 4 (scoring) is deterministic — no LLM call; pure function from signals to score
- Step 5 (hypothesis generation) prompt includes pathologies with score above threshold (0.4)

## TC-012 — Confirm/discard hypothesis

**Scope:** Hypothesis management  
**Spec ref:** §6.5, §4 (Hypothesis status enum)

### Preconditions
- Assessment status is `hypotheses_ready`
- At least one Hypothesis exists with status `pending`

### Main Flow (confirm)
1. Actor calls `PATCH /hypotheses/:id/confirm`
2. System updates Hypothesis status to `confirmed`
3. System returns updated Hypothesis

### Main Flow (discard)
1. Actor calls `PATCH /hypotheses/:id/discard`
2. System updates Hypothesis status to `discarded`
3. System returns updated Hypothesis

### Expected Outcomes
- `THEN` Only the targeted hypothesis is updated
- `THEN` A confirmed hypothesis is included in prescription generation
- `THEN` A discarded hypothesis is excluded from prescription generation
- `THEN` `PATCH /hypotheses/:id/confirm` is idempotent — confirming already-confirmed is a no-op (200 OK)

## TC-013 — Generate prescription (user-triggered)

**Scope:** Prescription generation  
**Spec ref:** §7 (step 7), §6.6, §4 (Prescription model)

### Preconditions
- Assessment status is `hypotheses_ready`
- At least one hypothesis is `confirmed`

### Main Flow
1. Actor calls `POST /assessments/:id/prescriptions/generate`
2. System executes pipeline step 7 (remedy recommendation)
3. LLM receives: confirmed hypotheses + applicable remedies from knowledge base
4. LLM returns `RemedyRecommendation[]` with rationale
5. System creates `Prescription` rows
6. System transitions assessment to `prescription_ready`
7. System returns `{ prescriptions: Prescription[] }`

### Expected Outcomes
- `THEN` Assessment status is `prescription_ready`
- `THEN` Each Prescription has: `remedyId`, `rationale`, `priority` (1–3), `estimatedEffort`
- `THEN` Prescriptions reference remedies from the knowledge base (not free-form invented remedies)
- `THEN` `GET /assessments/:id/prescriptions` returns the prescriptions

### Edge Cases
- No confirmed hypotheses → 422 `{ error: "No confirmed hypotheses", code: "NO_CONFIRMED_HYPOTHESES" }`

## TC-015 — Pipeline failure and recovery

**Scope:** AI Pipeline — error handling  
**Spec ref:** §7 (error semantics), §4 (status: failed), CLAUDE.md §Known Pitfalls

### Preconditions
- Assessment in `analyzing` status
- Pipeline step fails (simulated via mock)

### Main Flow
1. Pipeline step N throws an error
2. System catches error
3. System transitions assessment status to `failed`
4. System stores `failureReason` on assessment
5. System returns appropriate error response to subsequent status polls

### Expected Outcomes
- `THEN` Assessment status is `failed` in DB
- `THEN` `GET /assessments/:id/status` returns `{ status: "failed", failureReason: "..." }`
- `THEN` Failed assessment can be retried via `POST /assessments/:id/retry`

### Stale Recovery
- `THEN` On server startup, any assessment stuck in `analyzing` for > 10 minutes is reset to `evidence_pending` (not `failed`)
- `THEN` Startup recovery log entry exists
