## TC-007 — Upload assessment document

**Scope:** Document management  
**Spec ref:** §6.3, §8, §4 (EvidenceDocument model)

### Preconditions
- Actor is authenticated (Clerk session)
- Assessment exists with status `evidence_pending`
- Actor is a team member

### Main Flow
1. Actor selects a PDF file (≤ 10 MB)
2. Actor calls `POST /assessments/:id/documents` with `multipart/form-data`
3. System validates file type (pdf, md, txt) and size
4. System uploads file to Supabase Storage under `assessments/:id/`
5. System creates `EvidenceDocument` row (status: `uploaded`)
6. System triggers text extraction (async job)
7. System returns `{ id, fileName, status: "uploaded" }`

### Expected Outcomes
- `THEN` EvidenceDocument row exists in DB with correct assessmentId
- `THEN` File exists in Supabase bucket
- `THEN` Text extraction job is queued (document status moves to `extracted`)

### Edge Cases
- File > 10 MB → 413 with `{ error: "File too large", code: "FILE_TOO_LARGE" }`
- Unsupported file type → 400 with `{ error: "Unsupported file type", code: "INVALID_FILE_TYPE" }`
- Assessment not in `evidence_pending` → 422 with `{ error: "Assessment is not accepting evidence", code: "WRONG_ASSESSMENT_STATUS" }`

## TC-009 — Complete survey (manual answers)

**Scope:** Survey management  
**Spec ref:** §6.4, §4 (SurveyAnswer model), anti-double-counting rule

### Preconditions
- Actor is authenticated
- Assessment exists with status `evidence_pending`
- Assessment has at least one SurveyQuestion (seeded with knowledge base)

### Main Flow
1. Actor calls `GET /assessments/:id/survey` — gets list of questions
2. Actor calls `PUT /assessments/:id/survey/answers` with answers array
3. Each answer: `{ questionId, responseText, answeredBy: "manual" }`
4. System saves answers as `SurveyAnswer` rows
5. Actor calls `POST /assessments/:id/start` to advance to `analyzing`

### Expected Outcomes
- `THEN` All SurveyAnswer rows have `source: "manual"`
- `THEN` Manual answers DO qualify as independent `survey` source in scoring
- `THEN` Assessment status advances to `analyzing` on `POST /start`

### Anti-Double-Counting Verification
- `THEN` If any answer has `answeredBy: "ai_auto"` or `answeredBy: "ai_prefilled"`, its SurveyAnswer has `source: "document"` NOT `source: "survey"`
- `THEN` These answers do NOT add to survey-source score bonus in scoring engine

## TC-010 — Advance assessment to analyzing

**Scope:** Assessment lifecycle  
**Spec ref:** §4 (state machine §4.5), §6.1, §7

### Preconditions
- Actor is authenticated + team member
- Assessment exists with status `evidence_pending`
- At least one EvidenceDocument exists OR at least one manual SurveyAnswer exists

### Main Flow
1. Actor calls `POST /assessments/:id/start`
2. System validates minimum evidence requirement
3. System transitions assessment status to `analyzing`
4. System triggers async pipeline (steps 1–6)
5. System returns `{ status: "analyzing" }`

### Expected Outcomes
- `THEN` Assessment status is `analyzing` in DB
- `THEN` Background pipeline job is initiated
- `THEN` `GET /assessments/:id/status` returns `analyzing` with progress info

### Edge Cases
- No evidence at all → 422 `{ error: "No evidence available", code: "INSUFFICIENT_EVIDENCE" }`
- Assessment already `analyzing` → 409 `{ error: "Assessment already running", code: "ASSESSMENT_ALREADY_RUNNING" }`
- Assessment in `completed` → 422 `{ error: "Assessment already completed", code: "ASSESSMENT_COMPLETED" }`
