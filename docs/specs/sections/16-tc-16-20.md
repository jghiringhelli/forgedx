## TC-016 ‚Äî Generate final report

**Scope:** Report generation  
**Spec ref:** ¬ß9, ¬ß6.8, ¬ß4 (Report model)

### Preconditions
- Assessment status is `treatment_ready`
- TreatmentPlan exists

### Main Flow
1. Actor calls `POST /assessments/:id/report/generate`
2. System builds `ReportContent` JSONB from: assessment metadata, pathology scores, hypotheses, prescriptions, treatment plan
3. System generates PDF via Puppeteer
4. System uploads PDF to Supabase Storage
5. System creates `Report` row with `shareToken` (UUID)
6. System transitions assessment to `completed`
7. System returns `{ reportId, shareToken, pdfUrl }`

### Expected Outcomes
- `THEN` Assessment status is `completed`
- `THEN` Report row exists with `assessmentId`, `shareToken`, `pdfUrl`
- `THEN` `GET /assessments/:id/report` returns the report
- `THEN` ReportContent structure matches ¬ß9.1 JSON schema (gsScore, radarData, pathologies[], prescriptions[], timeline)

## TC-017 ‚Äî View shareable report (unauthenticated)

**Scope:** Report sharing  
**Spec ref:** ¬ß9.4, ¬ß6.8, ¬ß11 (funnel integration)

### Preconditions
- Report exists with known `shareToken`
- Actor is NOT authenticated (no Clerk session)

### Main Flow
1. Actor navigates to `/reports/:shareToken`
2. System validates shareToken
3. System returns read-only report view
4. Report includes: GS score, radar chart, pathology list, prescriptions
5. Report includes CTA: "Improve your GS score" ‚Üí links to Skool + workshop

### Expected Outcomes
- `THEN` Report page renders without authentication
- `THEN` No edit controls visible
- `THEN` GS score displayed with correct color semantics (0‚Äì4 red, 5‚Äì8 amber, 9‚Äì11 yellow-green, 12‚Äì14 green)
- `THEN` Radar chart renders with 7 axes (one per GS property)
- `THEN` Skool CTA button is visible with correct `href` = `NEXT_PUBLIC_SKOOL_COURSE_URL`
- `THEN` Workshop CTA button is visible with correct `href` = `NEXT_PUBLIC_WORKSHOP_URL`
- `THEN` Invalid shareToken ‚Üí 404 page

## TC-018 ‚Äî Public mini-assessment (funnel entry)

**Scope:** Funnel  
**Spec ref:** ¬ß11, ¬ß6.14, ¬ß4 (MiniAssessmentLead model)

### Preconditions
- Actor is NOT authenticated
- Actor is on the public landing page

### Main Flow
1. Actor completes 3-question mini-assessment (no auth required)
2. Actor calls `POST /funnel/mini-assessment` with 3 answers
3. System computes teaser score (deterministic, based on 3 answers)
4. System shows partial score + "Unlock full report" message
5. Actor optionally provides email (lead capture)
6. If email provided: system creates `MiniAssessmentLead` row
7. System shows Clerk auth CTA

### Expected Outcomes
- `THEN` `POST /funnel/mini-assessment` returns `{ teaserScore, teaserLabel, cta }` without authentication
- `THEN` teaserScore is between 0‚Äì3 (3-question subset of full 29-pathology scoring)
- `THEN` `MiniAssessmentLead` row created if email provided (optional)
- `THEN` Skool CTA visible even without email
- `THEN` Auth gate shows after teaser (Clerk sign-in/sign-up)

### Edge Cases
- Malformed answers ‚Üí 400 validation error
- Duplicate email ‚Üí upsert (no duplicate leads)

## TC-019 ‚Äî Funnel ‚Üí full assessment conversion

**Scope:** Funnel  
**Spec ref:** ¬ß11 (conversion flow), ¬ß6.14, ¬ß5 (auth)

### Preconditions
- Actor has completed mini-assessment (TC-018)
- Actor authenticates via Clerk (sign-up or sign-in)

### Main Flow
1. After Clerk auth, actor is redirected to `/dashboard`
2. System checks if actor has an existing project
3. If no project: onboarding wizard (create project ‚Üí create team)
4. Actor creates full assessment
5. System optionally pre-fills survey answers based on mini-assessment responses (`answeredBy: "ai_prefilled"`)
6. Actor arrives at full assessment flow

### Expected Outcomes
- `THEN` Pre-filled survey answers have `source: "document"` (NOT "survey") ‚Äî anti-double-counting
- `THEN` Actor can override pre-filled answers with manual responses (overrides set `source: "manual"`)
- `THEN` Mini-assessment lead record updated with `convertedAt` timestamp + `clerkUserId`

## TC-020 ‚Äî GS radar chart rendering

**Scope:** Frontend ‚Äî Report visualization  
**Spec ref:** ¬ß3 (design), ¬ß9 (radar data structure)

### Preconditions
- Report exists with valid `radarData` in ReportContent
- Actor views report (authenticated or via shareToken)

### Main Flow
1. Actor views report page
2. System renders radar chart component with 7 axes
3. Each axis: 0‚Äì2 scale (0 = absent, 1 = partial, 2 = strong)

### Expected Outcomes
- `THEN` Radar chart has exactly 7 axes: Self-Describing, Bounded, Verifiable, Defended, Auditable, Composable, Executable
- `THEN` Axis labels are PascalCase (matching GS white paper labels)
- `THEN` Each axis value is between 0 and 2 inclusive
- `THEN` Total GS score displayed = sum of all 7 axes (0‚Äì14)
- `THEN` Score color is forge-orange for 12‚Äì14, yellow-green for 9‚Äì11, amber for 5‚Äì8, red for 0‚Äì4
- `THEN` Radar chart is accessible (aria-label, role="img" with description)
